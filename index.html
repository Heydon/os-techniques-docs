HUIT "In Practice" Styleguide

This is a guide to writing and marking up 'techniques' examples of accessibility best practice for the Open Scholar system.

The title

The title for the Technique page should follow this format:

âœŽ Technique: [name of technique]

Container

Each “Technique” instance should be wrapped in the following manner, where [description] should describe the technique in a couple of sentences and [content] is the body of the technique.

<div class="os-example">
  <div class="os-example_description">
    <p>[description]</p>
  </div>
  [content]
</div>

Content

“Technique” instances are modular. A description for each is mandatory (see [description] above) and can constitute the entire instance or be followed by one or more subsections:

  • Example(s) (code or writing style examples)
  • Code editor (an external link to a codePen demo followed by an embedded version of that codePen)
  • Video: [video name] (A demonstration video, typically showing the technique as operated by a screen reader)
  • Download (a zipped package of reusable code or other resource)

Basic example

Here's a really simple instance which just uses the description field as its content:

<div class="os-example">
  <div class="os-example_description">
    <p>When writing copy, consider your audience and try not to alienate anyone. The key word is "short": Use short paragraphs with short sentences and, where possible, short words. Don't say anything that doesn't need to be said.</p>
  </div>
</div>

Supplementary subsections

Each of the optional subsections should start with an <h3>. This <h3> takes the os-expandable class, making it an expandable 'accordion' section with a supplied script. Hypothetically, we could extend the "Copy" example above with a download section:

<div class="os-example">
  <div class="os-example_description">
    <p>When writing copy, consider your audience and try not to alienate anyone. The key word is "short": Use short paragraphs with short sentences and, where possible, short words. Don't say anything that doesn't need to be said.</p>
  </div>
  <h3 class="os-expandable">Download</h3>
  <p><a href="path/to/writing-guide.pdf">How To Make Yourself Understood (pdf)</a></p>
</div>

(Note: Heading terminology for subsections should be consistent: Always one of "Example(s)", "Video: [video title]", "Download", "Code editor".)

Examples

When including an examples subsection, you have the option of including good and bad examples, indicated using "tick" and "cross" unicode symbols, like so:

<h3 class="os-expandable">Examples</h3>
<h4><span class="os-example_bad">&#x2717;</span> Bad example</h4>
<!-- describe the bad example -->
<h4><span class="os-example_good">&#x2713;</span> Good example</h4>
<!-- describe the good example -->

Code examples

To include a code block, you should wrap them in a <pre> tag containing an inline <code> tag. The 'less than' symbols of HTML examples should be encoded or the example will be treated as actual HTML.

<pre><code>
<img src="path/to/image.jpg" alt="Your alternative text."&gt;
</code></pre>

For inline code examples, just use <code><!-- code here --></code>.

Code editor

Some code — especially that with javascript functionality — is better sandboxed for the reader to explore and test. For this, we are using the jsBin service. The format is to include a link to the relevant jsBin URL.

<h3 class="os-expandable">Code editor</h3>
<p>A code editor is available to <a href="http://jsbin.com/rerecen/1/edit?html,css,output">explore and test three acceptable methods for employing icon fonts accessibly</a>.</p>

(Note: You can log in to jsBin and make code demos using harvardonlineaccessibility@gmail.com's Github account.)

Downloads

In some cases, it may be beneficial to attach a resource for the reader to download. This may be a document which describes or supplements the "in practice" instance or some library code for reuse. The structure should follow that below. Note the download attribute which improves the downloading interface in supporting browsers.

<h3 class="os-expandable">Download</h3>
<p><a href="path/to/a11yExpandable.zip" download>Expandable region jQuery plugin (zip)</a></p>

Video

In some cases, a video might be appropriate for clarifying something. For example, how something works in a screen reader. Videos must contain captions for the hard of hearing, as in this technique describing how to write captions.

The Video section should be titled "Video: [name of video]" where [name of video] is equal to the video's name on Youtube. For example: "Video: Writing closed captions". To upload and caption videos, use the harvardonlineaccessibility@gmail.com youtube account, which has the same password as the Github account.
